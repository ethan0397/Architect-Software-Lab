Software Architect Study Guide

A compact study site that brings together a roadmap, design patterns, architecture foundations, system design exercises, and a Hard Parts book track. The focus is on understanding problems, seeing trade-offs, and choosing the right structure for each system.

Learning map

Study in layers: code fundamentals, patterns, application architecture, distributed systems, then system design and decision making. Each section below is designed to make structure visible before depth.

Design Patterns

Learn patterns by problem type: object creation, module composition, and behavior coordination.

  • Factory, Builder, Singleton
  • Adapter, Facade, Decorator, Proxy
  • Strategy, State, Observer, Command

Architecture Foundations

See systems through boundaries, dependency direction, persistence, and event flow.

  • Layered, Modular Monolith, Hexagonal
  • Microservices and Event-Driven
  • NFRs, team ownership, cost of change

System Design

Move from requirements, NFRs, and scale to data flow, failure modes, and ADRs.

  • URL Shortener, Order Flow, Booking
  • Notification, Audit Log
  • Review checklist and portfolio format

Hard Parts Book Track

Study the book track to practice coupling, data ownership, distributed workflows, and trade-off analysis.

  • Modularity, decomposition, service granularity
  • Distributed access, sagas, contracts, data mesh
  • ADR, fitness functions, and decision mindset

Fundamentals Book Track

Study the book track to build foundations in architectural thinking, styles, risk, diagramming, and architect skills.

  • Modularity, characteristics, governance
  • Architecture styles and style selection
  • ADR, risk analysis, leadership

Six-month roadmap

Split into three rhythms: code-level design, application/system architecture, and architect mindset. The Hard Parts track deepens distributed architecture decisions from months 3 through 6.

Months 1-2

Code-Level Design

OOP, SOLID, cohesion, coupling, refactoring, unit tests, and core patterns.

Months 3-4

Application & Distributed Systems

Boundaries, modular monolith, hexagonal architecture, data consistency, retry, and timeout handling.

Months 5-6

Trade-offs & System Design

Choose architecture styles, design systems, evaluate NFRs, write ADRs, and explain decisions clearly.

How to study effectively

This site is strongest when you read, redraw, and then apply the ideas to a project of your own. Do not treat patterns like flashcards. Treat them as reusable responses to real problems.

Suggested reading order

  • Read the roadmap first to get the full picture.
  • Read design patterns to build code-level and module-level tools.
  • Read architecture foundations to understand structure and boundaries.
  • Work through system design exercises to build design reflexes.
  • Add the Hard Parts book track to practice tough distributed-architecture decisions.
  • Add the Fundamentals book track to build engineering-oriented architecture foundations.

How to self-check

  • Can you explain the topic with a real example?
  • Can you draw it clearly?
  • Can you name the trade-offs?
  • Do you know when not to use it?
