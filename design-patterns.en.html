Fast rule Do not ask whether a pattern is “good.” Ask what design problem you actually have, and then see whether a pattern makes the system simpler.

Creational patterns

Factory Method

Centralize object creation

Use it when implementation choice depends on config, environment, or input.

Builder

Create complex objects step by step

Helpful when objects have many optional fields or a multi-step construction flow.

Singleton

One shared instance

Useful for root configuration or logging, but easy to abuse into global state.

Structural patterns

Adapter

Integrate vendors safely

Normalize external interfaces so internal domain logic stays clean.

Facade

Hide multi-step flows

Turn a large use case into one clear entry point.

Decorator

Add logging, cache, or retry

Wrap behavior without modifying the original class.

Proxy

Control access

Useful for lazy loading, access control, or remote proxy scenarios.

Repository

Protect the domain from ORM details

Create a boundary between business logic and persistence concerns.

Behavioral patterns

Strategy

Multiple behavior variants

Turn long if/else branches into separate interchangeable strategies.

Observer

One event, many reactions

A publisher emits an event and subscribers handle their own responsibilities.

Command

Package actions as objects

Very useful for queues, retries, undo/redo, and deferred execution.

State

Behavior by lifecycle state

Great for order, booking, or payment state transitions.

Template Method

Fixed algorithm skeleton

Useful for import pipelines or workflows with the same structural steps.

Chain of Responsibility

Sequential processing stages

Fits middleware, validation chains, and approval flows.

Quick selection guide

If you notice...

  • Many if/else branches by behavior variant -> Strategy
  • Object creation logic scattered everywhere -> Factory Method
  • An object with too many constructor parameters -> Builder
  • Vendor APIs polluting your domain -> Adapter
  • A use case that calls 5-7 internal services -> Facade

Or if you notice...

  • You want cache, logging, or retry -> Decorator
  • An object changes behavior by lifecycle stage -> State
  • A request passes through many stages -> Chain of Responsibility
  • An action needs queueing or undo -> Command
  • Business logic is glued to ORM or SQL -> Repository
