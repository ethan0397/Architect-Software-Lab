Why this track matters

Many architecture resources teach concepts. This book teaches decision making when no perfect answer exists. It is especially strong for distributed architecture, where coupling, workflows, and data ownership constantly pull against each other.

Decision

Leave silver-bullet thinking behind

Stop asking which architecture is “best” and start asking which option best fits the current context.

Data

See data as an architectural force

Data ownership, transactions, and analytical flows strongly shape the way systems should be split.

Practice

Learn through real decisions

ADRs, workflows, sagas, granularity, and contracts are studied as operational architecture problems.

Book chapter map

Chapter group Focus Question to ask yourself
Chapter 1 Architecture without universal best practices How are context, data, and fitness functions shaping the decision?
Chapters 2-7 Pulling Things Apart Should I split by coupling, components, data domain, or service granularity?
Chapters 8-14 Putting Things Back Together After splitting, how should reuse, contracts, workflows, and data access be reassembled?
Chapter 15 Trade-Off Analysis Which alternative is better in this business, team, and operations context?
The key point Splitting a system is only half the story. The second half is accepting the new kinds of coupling that appear and managing them through ownership, contracts, workflows, and explicit trade-offs.

Eight learning modules

Module 1

Decision Mindset

Start with context, ADRs, fitness functions, and the mindset that there is no single best practice.

Module 2

Coupling and Architecture Quantum

Understand static coupling, dynamic coupling, and the parts that still need to be deployed together.

Module 3

Modularity and Decomposition

Learn component boundaries, decomposition drivers, and how to refactor a codebase with blurry seams.

Module 4

Data Decomposition and Service Granularity

Split data gradually, judge service size realistically, and notice where transactions glue things back together.

Module 5

Reuse, Data Ownership, and Distributed Access

Choose between shared libraries, shared services, sidecars, platforms, and cross-service data access patterns.

Module 6

Workflow, Orchestration, Choreography, and Sagas

Design distributed workflows with state, compensating actions, and explicit eventual consistency.

Module 7

Contracts and Analytical Data

Manage strict versus loose contracts, avoid stamp coupling, and separate operational and analytical flows.

Module 8

Trade-Off Analysis Lab

Pull everything together in an architecture review with alternatives, risks, and a clear conclusion.

Six-week schedule

Suggested pace

  • Week 1: chapters 1-2, write the first ADR and mark coupling points
  • Week 2: chapters 3-5, redraw the component map and domain boundaries
  • Week 3: chapters 6-7, split data domains and assess granularity
  • Week 4: chapters 8-10, choose reuse patterns and distributed data access
  • Week 5: chapters 11-13, build a workflow using choreography or sagas
  • Week 6: chapters 14-15, complete a full trade-off analysis
The most effective way to study it Read a chapter cluster, summarize it with a diagram, apply it to a real case in your head, and then write the decision as if you had to explain it to a team.

Applied labs

Case A

Order Processing

Case B

Internal Platform

Decide whether common logic should live in a library, shared service, sidecar, or platform capability.

Case C

Booking System

Signals that the track clicked

  • You can name two or three competing trade-off dimensions instead of one preferred solution.
  • You can explain why an imperfect boundary is still acceptable in a given context.
  • You can tell the difference between splitting for lower coupling and splitting for appearance.
  • You can write a short ADR that teammates immediately understand.

How it connects to the rest of the site

  • Go back to Design Patterns when you need component-level or integration-level solutions.
